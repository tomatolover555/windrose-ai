## Start with a machine-readable contract

For agents, documentation is not just prose. It is a contract that must be precise enough to drive code generation, validation, and recovery.

At minimum, your schema should specify:

- required versus optional fields
- allowed values, formats, and regex constraints
- defaults and whether they are server-side or client-side
- pagination style and cursor semantics
- authentication scopes per operation
- rate limits and retry guidance
- error codes and retryability
- side effects and whether they are immediate or deferred

If you publish OpenAPI, do not stop at endpoint names and request bodies. Include:

- `operationId` for every action
- explicit `requestBody` schemas
- response schemas for every status code you expect clients to handle
- `oneOf` / `anyOf` only when the branching is truly necessary
- examples for success and failure cases
- `429`, `409`, and `422` responses when they matter
- links or callbacks if the workflow is asynchronous

If you use JSON Schema directly, be equally strict:

- distinguish `required` from `nullable`
- define enums instead of relying on prose
- specify `minLength`, `maxLength`, `minimum`, `maximum`, and `pattern` where applicable
- avoid “stringly typed” fields like `status`, `type`, or `mode` unless the allowed values are enumerated

A human can often recover from vague docs by asking in Slack or reading source code. An agent cannot. It needs a structured source of truth it can parse, validate against, and plan from.

## Make errors actionable, not decorative

A lot of APIs return errors that are technically correct and practically useless.

“Invalid request.”
“Something went wrong.”
“Forbidden.”

For an agent, that is not just frustrating — it breaks recovery. The agent needs to know whether to fix input, refresh credentials, wait and retry, or stop entirely.

A useful error response should include:

- a stable machine-readable code
- an HTTP status that matches the failure class
- a short human-readable message
- the field or subresource that failed
- whether the error is retryable
- a remediation hint when the fix is known

For example:

```json
{
  "error": {
    "code": "quantity_out_of_range",
    "message": "quantity must be between 1 and 100",
    "field": "quantity",
    "retryable": false,
    "hint": "Reduce quantity or split the order into multiple requests."
  }
}
```

That gives the agent enough information to choose a next step:

- `retryable: false` means do not resend unchanged
- `field: quantity` tells it where to patch the request
- `hint` gives it a concrete repair strategy

For authentication failures, be similarly explicit. A `401` should usually mean the token is missing or invalid; a `403` should mean the token is valid but lacks permission. If the token is expired, say so. If the user must re-consent to a scope, say that too.

The goal is not prettier errors. The goal is errors that let an agent classify the failure and recover without guessing.

## Declare capabilities structurally

An agent should not have to infer what your API can do by reading examples and guessing from endpoints.

Instead, publish a capability document that answers these questions per action:

- What does this operation do?
- What preconditions must be true?
- What inputs are required?
- What side effects occur?
- Is it safe, idempotent, or irreversible?
- What is the expected latency or completion model?
- What failures are common and which are retryable?

If you use OpenAPI, make these properties explicit in the operation description and schema metadata. If you use a separate manifest, include fields like:

- `safe: true|false`
- `idempotent: true|false`
- `requires_confirmation: true|false`
- `side_effects: ["charge_card", "reserve_inventory"]`
- `preconditions: ["customer_has_payment_method"]`
- `completion: "synchronous" | "async"`

A concrete example matters here. `POST /orders` is not enough information by itself. An agent needs to know whether that endpoint:

- creates a draft only
- reserves inventory immediately
- charges the customer immediately
- requires a shipping address first
- returns a job ID for later polling

Those differences change the plan the agent should execute.

If you can, publish a machine-readable “capabilities index” alongside your API docs so an agent can discover available actions without scraping prose. The point is not just discoverability — it is reducing ambiguity before the first request is made.

## Idempotency is not optional for agentic workflows

Agents retry more often than humans, and they often retry without perfect knowledge of whether the first attempt succeeded.

That makes idempotency essential for any endpoint that can create duplicate side effects.

Use idempotency keys on operations such as:

- creating orders
- charging cards
- sending messages
- submitting jobs
- reserving inventory
- issuing refunds
- changing subscription state

A good implementation should define:

- how long the key is retained
- whether the same key with a different payload is rejected
- whether the response is replayed verbatim or reconstructed
- what happens if the first request is still in flight
- whether idempotency applies across regions or only within one shard

For example, if a client sends:

```http
POST /charges
Idempotency-Key: 8f3d2c1a-7b6e-4d1f-9c2a-1b0e5f4d9a11
```

and the connection drops after the server processes it, a retry with the same key should return the original result, not create a second charge.

That detail matters because agents often operate in uncertain execution environments: tool calls time out, model chains repeat steps, and orchestration layers may replay requests after transient failures.

Not every endpoint needs idempotency. Read-only requests usually do not. But if a duplicate request would cost money, send duplicate notifications, or mutate state twice, idempotency should be part of the contract — not an implementation detail.

## Use quote-before-commit for meaningful actions

For any action with cost, irreversible impact, or user-visible consequences, separate estimation from execution.

A useful quote-before-commit flow should return more than just a price. It should include the exact terms the agent needs to decide whether to proceed:

- total cost, including taxes and fees
- currency
- expiration time for the quote
- delivery or completion estimate
- cancellation or refund rules
- resource usage or quota impact
- the exact payload that will be committed, if it differs from the draft

A concrete pattern looks like this:

1. `POST /shipments/quote`
2. server returns quote ID, price, ETA, and constraints
3. `POST /shipments/commit` with the quote ID
4. server verifies the quote is still valid and executes the action

This matters because agents can compare quotes, ask for approval, or stop before committing if the result is too expensive or risky.

It is especially useful for:

- shipping labels
- ad spend
- cloud resource provisioning
- purchases
- publishing content
- destructive admin actions

The key implementation detail is that the commit step should bind to the quote. If the user changes the address, quantity, or destination after quoting, the server should reject the commit or require a new quote. Otherwise the “quote” is just a suggestion, not a safeguard.

## Schema clarity is a product feature

Schema clarity sounds boring. It is also one of the highest-leverage improvements you can make.

Agents fail when schemas are ambiguous in ways humans can usually repair by intuition. Common problems include:

- overloaded fields
- inconsistent naming
- undocumented defaults
- mixed data types
- nullable fields with unclear meaning
- arrays whose item order matters but is not stated
- timestamps without timezone rules

A good schema removes ambiguity before execution starts.

Practical rules:

- use one name for one concept
- prefer explicit enums over free-form strings
- document units, currency, and timezone assumptions
- distinguish omitted, null, and empty values
- make nested objects predictable
- avoid “magic” behavior unless it is clearly documented

A few examples of the kind of precision that helps agents:

- `amount_cents` is better than `amount` if the unit is fixed
- `status: ["draft", "quoted", "committed", "failed"]` is better than `status: "active"`
- `expires_at` should specify UTC or an offset, not “local time”
- `phone_number` should define whether E.164 is required
- `tags: []` should mean “intentionally empty,” while omission should mean “not provided”

Also document what happens when fields conflict. For example, if both `discount_code` and `manual_discount_percent` are present, which one wins? If `shipping_speed` is omitted, what default applies? If `metadata` exceeds a size limit, does the request fail or truncate?

If your API is used by both humans and agents, clarity helps both. Humans make fewer mistakes. Agents need fewer corrective loops.
