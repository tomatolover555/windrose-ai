## Write for decision-making, not just documentation

A good OpenAPI spec for agents should help the model make a call decision, not just understand the endpoint in isolation. In practice, that means each operation should answer:

- **When is this the right tool?**
- **What must already be true before calling it?**
- **What state changes if it succeeds?**
- **What failure modes are expected, and which are recoverable?**

For example, compare these two descriptions:

- “Creates a user.”
- “Creates a team member account for an organization. Use only after the organization has been verified and the inviter has confirmed the email address. This operation sends an invitation email and returns a pending membership record.”

The second version gives the agent enough context to avoid premature calls, duplicate invitations, and incorrect assumptions about side effects. It also tells the model what success means: not just “a user exists,” but “an invitation was created and the membership is pending.”

A spec that only describes shape is incomplete for agents. The model also needs decision cues.

## Description quality matters more than people expect

Descriptions are the main place where you can encode intent that schemas cannot express.

A useful description should usually include:

- the action in plain language
- the resource being changed or retrieved
- preconditions or required state
- side effects
- constraints that are not obvious from the schema
- the difference between similar operations

For example, if you have both `archiveOrder` and `deleteOrder`, the descriptions should say whether the order remains searchable, whether it can be restored, and whether related invoices or shipments are affected. If `archiveOrder` is reversible but `deleteOrder` is not, say that explicitly.

Avoid descriptions that sound complete but do not help with tool choice:

- “This endpoint manages orders”
- “Use this to interact with inventory”
- “Returns relevant information”

Those phrases do not tell an agent when to use the operation, what it changes, or what makes it different from nearby endpoints.

A good test: if you removed the operation name, would the description still let an agent choose correctly between similar tools? If not, it is too vague.

## Examples are not optional

Examples are one of the fastest ways to reduce ambiguity for agents because they show the exact shape of a valid request or response, not just the abstract schema.

At minimum, include examples for:

- a typical request
- a minimal valid request
- a request that uses an optional field
- a boundary case
- a successful response
- a failure response that the agent might need to recover from

For example, if an endpoint accepts either a `customer_id` or an email address, show both forms in separate examples. If a timestamp must be UTC ISO 8601, show the exact string format, including the trailing `Z`. If pagination uses cursors, show a realistic cursor value and how it appears in the response.

Good examples should also reflect real workflow, not just isolated payloads. If creating a resource is usually followed by a status check, show the creation response in a way that makes the next call obvious.

Bad examples create false confidence. If the example omits a required field, uses placeholder names that do not match the schema, or shows a value that is technically valid but operationally misleading, the agent may copy the wrong pattern.

## Error semantics should be explicit and machine-usable

For agents, error handling is not just about showing a message. It is about deciding what to do next.

A useful error contract should tell the agent whether the failure is:

- a malformed request
- a validation problem the agent can fix
- an authentication or authorization issue
- a conflict with current state
- a rate limit
- a transient server or dependency failure

Status codes help, but they are not enough on their own. Add a structured error body such as:

```json
{
  "error_code": "payment_method_expired",
  "message": "The saved payment method has expired.",
  "field": "payment_method_id",
  "retryable": false,
  "details": {
    "required_action": "provide_new_payment_method"
  }
}
```

That gives the agent enough information to stop retrying the same request and instead ask the user for a new payment method.

A good error model should also distinguish between:
- **retryable** failures, where the same request may succeed later
- **repairable** failures, where the agent can fix input and try again
- **terminal** failures, where the agent should escalate or stop

If you support rate limiting, include a `retry_after_seconds` field or equivalent. If a request is idempotent, say so in the operation description or response semantics. That helps agents avoid duplicate side effects.

## operationId is a navigation tool, not just a label

For agents, `operationId` is not just a codegen convenience. It is a stable handle the model can use to distinguish one action from another.

Good `operationId` values are:

- **stable** — they do not change unless the operation meaning changes
- **unique** — one operation, one identifier
- **descriptive** — the name suggests the action
- **action-oriented** — it starts with a verb

A practical naming pattern is:

- `listOrders`
- `getOrder`
- `createOrder`
- `cancelOrder`
- `refundOrder`

When there are multiple variants, add a qualifier that reflects the real difference:

- `createDraftInvoice`
- `finalizeInvoice`
- `sendInvoice`
- `archiveOrder`
- `deleteOrder`

Avoid names that hide intent or collapse distinct actions into one bucket:

- `manageUser`
- `doThing`
- `endpoint42`
- `createOrUpdateWhatever`

If two operations are easy to confuse, the `operationId` should make the difference obvious without reading the full spec. That helps both tool selection and downstream routing logic.

Also, do not rename `operationId` values casually. In agentic systems, changing them can break prompts, evals, cached tool mappings, and routing heuristics even if the HTTP path stays the same.

## Structure helps agents, but only if it reflects reality

OpenAPI structure is useful only when it matches how the API actually behaves.

Use schemas, enums, required fields, and response codes to make the contract precise. But do not rely on structure to communicate things that the schema cannot express, such as:

- when an operation should be used
- whether a failure is recoverable
- whether a resource is soft-deleted or permanently removed
- whether a field is optional in theory but required in practice for a common workflow

A precise schema with a vague description still leaves the agent guessing. A detailed description with no examples still leaves room for misinterpretation. A well-typed error response with no retry semantics still leaves the model unsure whether to try again.

The strongest specs align all three:
- the **schema** defines the shape
- the **description** defines the intent
- the **examples** show the actual usage pattern

That alignment is what reduces uncertainty for both humans and agents.
