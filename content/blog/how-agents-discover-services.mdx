## Start with the public surface

The first thing an agent usually encounters is not an API. It is a homepage, a docs site, a sitemap, or a search result snippet. That matters because these surfaces often contain the only clues an agent has before it decides whether deeper discovery is worth the cost.

A useful site makes the transition from human-facing to machine-facing content explicit. In practice, that means exposing:

- a homepage that names the product and primary use cases,
- a docs page with stable URLs,
- a `llms.txt` file that points to the most relevant docs and policies,
- a `/.well-known/` document that identifies canonical metadata,
- and an OpenAPI spec or equivalent contract for actual execution.

The key detail is ordering. Agents should not have to infer where the “real” interface lives by crawling every page. If the homepage says “developers,” the docs should be one click away. If the docs say “API,” the spec should be linked directly from there. If the service supports automation, the public surface should make that path obvious instead of hiding it behind marketing copy.

## Agent store manifests: curated discovery

Some services are not meant to be found by crawling at all. They are discovered through a registry entry in an agent store, app marketplace, or partner directory. In those systems, the manifest is the first authoritative description an agent sees.

A useful manifest usually includes more than a name and logo. It should tell the agent:

- the exact action categories supported, such as “create invoice,” “check inventory,” or “book appointment,”
- the auth model, such as OAuth, API key, or delegated user consent,
- whether the action is read-only or state-changing,
- any quota, region, or plan restrictions,
- and the canonical URL for the API or docs.

That level of detail matters because marketplace discovery often happens before the agent has any context about the service. A manifest that only says “connect your store” forces the agent to guess what “connect” means. A manifest that says “read inventory, create draft orders, and update fulfillment status” gives the agent a real capability boundary.

The tradeoff is portability. A manifest in one ecosystem may not be recognized anywhere else, so it is best treated as a distribution channel, not the source of truth. The source of truth should still be the API spec and the service’s own policy docs.

## How the agent gets from zero knowledge to checkout

A realistic agent flow is less “browse and buy” and more “discover, verify, constrain, then act.”

1. **Start from a seed**
   - The seed may be a domain name from the user, a marketplace listing, or a search result.
   - If the user says “order from Acme Supplies,” the agent should begin with that domain, not a generic web search.

2. **Resolve the discovery layer**
   - Check `/.well-known/` for metadata such as API pointers, auth endpoints, or policy documents.
   - Read `llms.txt` to find the docs, pricing, and support pages most likely to matter.

3. **Load the contract**
   - Fetch the OpenAPI spec or equivalent machine-readable description.
   - Confirm the exact operation name, required fields, and error responses before attempting anything.

4. **Check whether the action is allowed**
   - Verify the user’s account tier, region, and permissions.
   - Look for explicit constraints like “draft only,” “requires manual approval,” or “not available for guest accounts.”

5. **Authenticate with the declared mechanism**
   - Use the auth flow the service actually documents.
   - If the spec says OAuth with offline access, do not improvise with a session cookie or browser login.

6. **Create a reversible draft if possible**
   - Prefer a quote, cart, draft order, or preview endpoint before a final commit.
   - This gives the agent a chance to validate totals, shipping, taxes, and item availability.

7. **Commit only after validation**
   - Submit the final action only when the response matches the expected schema and the user’s intent is still clear.
   - Store the confirmation ID, receipt URL, or transaction reference for auditability.

That flow is important because the failure mode is usually not “the agent couldn’t find the site.” It is “the agent found a path that looked valid but skipped the checks that make the action safe.”

## Practical guidance for developers

If you want agents to use your service reliably, optimize for three things: discoverability, contract quality, and safe execution.

- **Discoverability:** publish `llms.txt`, a `/.well-known/` document, and stable docs URLs.
- **Contract quality:** keep your OpenAPI spec complete, current, and aligned with production behavior.
- **Safe execution:** document which actions are reversible, which require confirmation, and which are blocked by policy or account tier.

A few concrete implementation details help a lot:

- include example requests and responses for every write operation,
- document rate limits and idempotency behavior,
- mark deprecated endpoints clearly,
- and keep auth scopes aligned with the smallest useful permission set.

If your service has multiple surfaces, make one canonical path and point everything else to it. Agents do better when there is one authoritative spec, one canonical docs location, and one place to check policy.
