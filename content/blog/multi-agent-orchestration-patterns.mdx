## The Core Question: Coordination

Before choosing a framework, define the dependency graph of the task. In practice, most multi-agent systems fall into one of three shapes:

- **Linear dependency**: the next step cannot start until the previous step produces a specific artifact.
  - Example: extract fields from a contract, then validate them against a schema, then generate a summary.
- **Fan-out / fan-in**: several subtasks can run independently, then a reducer combines them.
  - Example: query three knowledge sources in parallel, then reconcile the results.
- **Supervisor / worker**: one agent decides what work to delegate, then checks whether the result is acceptable.
  - Example: a manager agent assigns code review, test generation, and documentation updates to specialist agents.

This matters because orchestration failures usually come from choosing the wrong shape, not from choosing the wrong model. A task that looks “agentic” may actually be a deterministic pipeline with one or two validation steps. In those cases, adding a second or third agent often increases latency and failure rate without improving output quality.

A useful rule: if you can describe the system as “produce artifact A, then transform it into B, then verify C,” you probably want explicit workflow orchestration, not open-ended collaboration.

## Sequential Orchestration

Sequential orchestration is a chain of steps where each stage produces a structured artifact for the next stage.

A concrete example:

1. **Retriever agent** pulls 5 relevant documents.
2. **Extractor agent** converts them into a JSON object with fields like `claim`, `source`, and `confidence`.
3. **Verifier agent** checks whether each claim is supported by at least one source.
4. **Writer agent** turns the verified claims into prose.

This pattern works because each step has a narrow contract. The retriever does not need to write. The writer does not need to search. The verifier does not need to invent new claims.

### Strengths
- Easy to inspect intermediate outputs
- Easy to retry only the failed step
- Works well with schema validation and typed outputs
- Good fit for compliance, ETL-like tasks, and report generation

### Pitfalls
- A bad early extraction can poison every downstream step
- Each handoff adds latency and token cost
- If the contract between steps is vague, the chain becomes fragile

A practical implementation detail: define the output schema before you define the prompt. For example, if the next step expects `{"decision": "approve|reject", "reason": "...", "evidence_ids": [...]}`, make that contract explicit and reject anything else. This is one of the simplest ways to keep a multi-step system debuggable.

LangGraph is especially strong here because it makes state transitions explicit. You can persist the current node, inspect the state after each edge, and branch based on concrete conditions instead of hoping the model “remembers” what happened earlier.

## Parallel Orchestration

Parallel orchestration is useful when the subtasks are independent and the main goal is to reduce wall-clock time or improve coverage.

A concrete example:

- run the same question against three search strategies
- ask two different models to critique a draft
- extract entities from 20 documents in parallel
- evaluate a code change with unit tests, static analysis, and a security check at the same time

The key design problem is not fan-out. It is **reconciliation**.

### Strengths
- Faster completion for independent work
- Better coverage when one branch may miss something
- Useful for cross-checking and ensemble-style evaluation

### Pitfalls
- The merge step can become the real bottleneck
- Branches may disagree in ways that are hard to resolve automatically
- Costs can spike if you parallelize work that was never truly independent

A good parallel system usually has a deterministic reducer. For example:
- choose the highest-confidence answer only if it is supported by at least two branches
- merge extracted entities by ID, then deduplicate by source
- fail closed if branches disagree on a safety-critical decision

Without a clear reducer, parallel agents just create more text for another model to sort through. That is not orchestration; it is deferred confusion.

AutoGen can help when the branches need to critique or refine each other, but it is still important to separate the “discussion” layer from the “decision” layer. If you cannot explain how the final answer is selected, the system is too loose for production.

## Hierarchical Orchestration

Hierarchical orchestration is most useful when the task is too large for one agent but still needs a single point of control.

A concrete pattern:

- the supervisor receives the user request
- it decomposes the request into 3–5 bounded tasks
- each worker returns a structured result
- the supervisor checks for gaps, conflicts, and missing evidence
- if needed, it sends one targeted follow-up task instead of restarting the whole workflow

This is often the right shape for research, code review, and multi-step planning. It gives you a place to enforce quality without requiring every worker to understand the whole problem.

### Strengths
- Good for tasks with changing requirements
- Lets you add specialists without redesigning the whole system
- Makes it easier to insert human review at the top level

### Pitfalls
- The supervisor can become a throughput bottleneck
- Poor decomposition leads to duplicated work
- If the supervisor is too permissive, it can spawn unnecessary subagents and inflate cost

A practical guardrail is to cap delegation depth and task count. For example, a supervisor might be allowed to create at most 4 subtasks and only one additional round of follow-up. That keeps the system from turning a simple request into a recursive planning loop.

CrewAI maps naturally to this pattern because it encourages role definitions and task assignment. That is useful for teams that think in terms of “researcher,” “analyst,” and “writer,” but you still need explicit logging and state if you want to understand why the crew produced a particular result.

## LangGraph, AutoGen, and CrewAI: Honest Assessment

### LangGraph
LangGraph is strongest when you need a workflow you can inspect, replay, and control.

Best fit:
- stateful pipelines
- branching based on intermediate outputs
- human approval steps
- audit trails and retries

Why teams choose it:
- explicit graph structure
- easier to reason about than free-form agent chat
- better fit for production debugging

Tradeoff:
- you have to design the graph up front
- it can feel heavier for small experiments

### AutoGen
AutoGen is strongest when the interaction between agents is part of the solution.

Best fit:
- debate-style refinement
- collaborative coding
- iterative critique and revision
- experiments where agent conversation is the mechanism, not just the transport

Why teams choose it:
- easy to express multi-agent dialogue
- good for exploratory workflows
- useful when one agent’s output should directly influence another’s reasoning

Tradeoff:
- conversation can drift without strong constraints
- harder to make deterministic
- less natural for strict workflow-style systems

### CrewAI
CrewAI is strongest when you want a quick role-based abstraction.

Best fit:
- small internal tools
- demos
- simple task decomposition that mirrors human roles

Why teams choose it:
- easy mental model
- fast to prototype
- low setup overhead

Tradeoff:
- the abstraction can hide important execution details
- branching and recovery are less explicit
- it becomes harder to debug as the system grows

The practical distinction is this: LangGraph helps you control execution, AutoGen helps you coordinate conversation, and CrewAI helps you assign roles quickly. None of them removes the need for good task design, structured outputs, and observability.

## What Actually Works at Scale

At scale, the systems that survive usually have these properties:

1. **A single-agent baseline first**
   - Measure whether one model plus tools can solve 80% of the task before adding orchestration.
2. **Explicit contracts between steps**
   - Use schemas, typed outputs, and validation rules for every handoff.
3. **Bounded fan-out**
   - Limit how many branches or subtasks can be created from one request.
4. **Deterministic reducers**
   - Define exactly how parallel outputs are merged or rejected.
5. **Step-level observability**
   - Log prompts, outputs, tool calls, latency, token usage, and failure reasons per step.
6. **Recovery paths**
   - Retry only the failed node, not the whole workflow, and preserve intermediate state.
7. **Human intervention points**
   - Add approval gates for high-risk or ambiguous decisions.

The pattern that shows up again and again is that the best multi-agent systems are not maximally autonomous. They are tightly bounded systems with clear state transitions and narrow responsibilities. The more expensive the mistake, the less freedom you should give the agents.
